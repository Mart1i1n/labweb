Web3 登录与盲消息攻击演示

Web3 登录通常基于钱包地址和签名。前端负责交互，后端负责生成消息与校验签名，钱包负责展示消息并完成签名；只有后端验证通过后，用户才能访问站内的 off-chain 数据。

前端负责交互与传递。
后端负责生成消息、校验签名并决定登录结果。
钱包负责签名，也决定用户能否看清消息究竟将用于哪个网站。

本页面展示 Web3 登录的基本机制、Blind Message Attack 的完整链路，以及一种基于钱包检测域名不一致的缓解思路。

对 27 个知名网站的检测显示，75.8% 的 Web3 登录存在不同程度的盲消息攻击风险，影响用户超过 200 万。为降低这类风险，本文在 MetaMask 中实现了 Web3AuthGuard，可在登录签名前自动检查消息来源，并提醒用户可疑的 Web3 登录；评估结果显示，该工具成功检测出 80% 的潜在攻击。

盲消息攻击 Blind Message Attack

用户以为自己只是在当前网站完成一次普通登录，但恶意网站实际向用户展示的是“另一个目标网站”的登录消息。由于用户或钱包难以判断该消息真正服务于哪个网站，用户会盲目签名；攻击者拿到签名后，再把它提交给目标网站后端，最终在目标网站完成未授权登录并访问用户数据。

核心问题：用户签的是消息，但很难确认这个消息究竟属于哪个网站。
问题不在于签名算法失效，而在于“签名内容与来源之间缺乏清晰绑定”，导致用户在恶意网站上完成的签名，能够被转用于另一个站点的后端认证。
案例二：盲消息攻击链路

用户访问恶意网站 M。恶意网站伪装成普通登录页面，但实际上诱导用户签署的是“网站 B 将接受的登录消息”。用户签名后，恶意网站把地址、消息、签名转发给网站 B 后端，由网站 B 后端完成认证，最终导致网站 B 登录成功（未授权）并开放数据访问。

缓解思路：消息绑定域名 + 钱包侧一致性检查

如果登录消息中明确包含目标域名，钱包就可以在签名中检查该域名，把它与当前访问页面域名进行对比。一旦发现不一致，钱包就直接告警并中止签名。

消息中明确写入目标域名，例如 Domain: websiteB.com。
钱包高亮显示该域名，避免用户忽略。
钱包同时展示当前访问页面域名并执行比对。
若域名不一致，则直接阻止签名。
案例三：基于钱包的安全防护流程

用户仍然访问恶意网站 M。恶意网站尝试诱导用户签署网站 B 的登录消息，但这次消息中明确包含 websiteB.com。钱包识别该域名，再把它与当前页面 malicious-site.com 比较；由于两者不一致，钱包告警并阻止签名，攻击在签名前被拦截。
