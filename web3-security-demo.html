Web3 登录盲消息风险演示

Web3 登录通常基于钱包地址与签名。前端发起交互，后端生成并校验消息，钱包负责展示与签名；校验通过后，用户才能访问站内 off-chain 数据。

前端发起登录请求。
后端生成消息并校验签名。
钱包展示消息并完成签名。

本页面展示 Web3 登录基本流程、盲消息风险链路，以及基于域名校验的缓解方案。

对 27 个知名网站的检测显示，75.8% 的 Web3 登录存在盲消息风险，相关用户规模超过 200 万。为降低风险，本工作实现了 Web3AuthGuard，可在签名前检查消息来源；评估显示可识别 80% 的潜在风险场景。

盲消息安全风险 Blind Message Attack

用户以为自己在当前网站完成普通登录，但诱导站点实际展示的是“另一个目标网站”的登录消息。由于消息来源没有和网站绑定，签名可能被用于登录目标网站，带来跨站数据访问/泄露风险。

核心问题：用户签名的消息没有和登录的网站绑定。
问题不在签名算法本身，而在于“签名内容与来源缺乏清晰绑定”，导致签名可能被转用于另一个站点的后端认证。
案例二：盲消息风险链路模拟

用户访问诱导站点 M。该站点展示的是“网站 B 的登录消息”。用户签名后，签名内容由站点 M 提交给网站 B ，从而实现未授权访问用户在网站 B 的数据，造成数据泄露。

缓解思路：消息绑定域名 + 钱包侧一致性检查

如果登录消息中明确包含目标域名，钱包就可以将其与当前页面域名比对；一旦不一致，即可告警并中止签名。

消息中写入目标域名，例如 Domain: websiteB.com。
钱包高亮显示目标域名。
钱包同时展示当前页面域名并进行比对。
域名不一致时直接阻止签名。
