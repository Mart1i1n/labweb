Web3组件的中心化风险分析

基于钱包与链上合约的大规模分析，中心化风险并非孤立现象，而是在部分 Web3 基础设施中具有一定普遍性。

对 28 款以太坊官方推荐钱包的深入分析显示，27 款存在不同程度的中心化风险；对超过 11 万份以太坊链上合约的自动化检测显示，83.5% 存在不同程度的中心化风险，其中包括 260 个总市值超过 980 亿美元的高价值合约。

用户通常只看到标准 Web3 交互链路；但当钱包连接注册信息、备份服务、RPC 或第三方服务时，钱包可能成为数据汇聚与传播节点。与此同时，若 DApp 依赖的 Smart Contract 保留过强的 owner 权限，控制能力也可能重新集中。

SR#1 Anonymity Loss

去匿名化

一些加密钱包要求用户注册，并在注册过程中收集个人身份信息（Personally Identifiable Information, PII），如电子邮件地址和电话号码。例如，Bitcoin.com Wallet 是一款知名的加密钱包，下载量超过 500 万次，它提供可选的注册功能。类似地，一些 Web3 网站支持社交登录，如 Google 和 Facebook，这些中心化身份认证方式可能将用户身份信息与其公钥建立关联，从而削弱公钥带来的匿名性。此外，个人身份信息一旦泄露，也可能进一步放大钓鱼等安全风险。例如，Trezor 钱包曾泄露 106,856 名用户的电子邮件地址，并引发了后续的钓鱼邮件事件。

SR#2 Private Key Leakage

私钥泄露

私钥是 Web3 生态的核心，用于管理用户的加密资产，必须安全存储在本地设备上。然而，一些钱包鼓励用户将私钥备份到第一方或第三方服务器。例如，Bitcoin.com Wallet 在设置页面提供了私钥备份选项，允许用户将私钥上传至 Bitcoin.com Wallet 的服务器。另一个钱包 ZenGo 要求用户在使用前必须将私钥备份至 Google Drive。这种做法可能导致私钥在传输或存储过程中暴露。例如，若用户通过不安全的网络传输私钥，相关数据可能被中间人截获并造成私钥泄露。一个真实案例是 Slope 钱包在传输过程中泄露用户私钥，导致约 600 万美元的代币被盗。

SR#3 Built-in Service

内置服务

许多加密钱包内置中心化服务，例如，Trust Wallet 内置了 Binance 交易所，以提供加密货币的购买服务。这些中心化服务通常受到监管，并可能要求用户提供身份信息。普通用户往往难以区分钱包内的中心化与去中心化服务，因此在用户使用相关服务时，钱包应明确告知相应风险。这些服务也可能因监管或运营问题导致宕机或停止运营，进而给用户带来资产损失风险。例如，近年来已有 75 家交易所停止运营，其中 Celsius Network 拥有 170 万名用户，其暂停服务曾造成大量用户的经济损失。

SR#4 RPC Services

RPC 服务

加密钱包依赖 RPC 服务与区块链交互，用于查询链上数据或提交交易。然而，第三方 RPC 服务存在固有的中心化风险，例如服务不可用风险。此外，RPC 服务提供商可以提前访问用户的交易信息，从而可能引发 MEV 相关的交易排序与公平性风险。因此，钱包应披露默认的 RPC 服务提供商，并允许用户自定义 RPC 服务。RPC 服务的中心化已导致多起安全问题，例如，2022 年 3 月，Infura 因政策原因切断了对乌克兰用户的服务；由于钱包 MetaMask 的默认 RPC 服务商是 Infura，导致大量用户受到影响，无法访问区块链。

SR#5 Third-party SDKs

第三方 SDK

加密钱包通常依赖第三方 SDK 实现通知、数据分析等功能，但在此过程中，钱包可能会向第三方服务提供商共享用户的 PII、设备 ID 和崩溃日志等数据。例如，在 Slope 事件中，白帽研究人员发现 Slope 钱包会将日志以明文形式传输至第三方漏洞分析服务商 Sentry，其中包含用户大量敏感信息。

Limited Liquidity

流动性限制

在加密货币市场中，流动性指代币的自由交易能力。本文将流动性定义为：任何人均可自由转移（transfer）其持有的代币。如果合约所有者可以控制代币转账相关的函数，则认为该合约存在流动性限制风险。

Vulnerable Scarcity

稀缺性降低

加密资产的价值通常受供应量影响。一些 token 或 NFT 的供应量是固定的，由于稀缺性，它们的市场价值也较高。然而，若 Owner 可随意增发代币，会导致代币稀缺性降低，从而损害现有持有者的利益。一个现实案例是 Yearn.finance 团队曾提议额外发行价值 2.25 亿美元的 YFI 代币以激励开发者，此举引发持有者强烈反对。

Mutable Metadata

可变元数据

ERC20代币合约通常提供一组可选元数据，以标识代币的基本信息，包括名称（name）、符号（symbol）和小数位数（decimals）。这些元数据通常在合约部署后保持不变。然而，若Owner可以随意修改元数据，则可能导致用户误解代币属性，甚至影响市场交易。例如 ERC20代币如果合约允许Owner修改这些字段，则可能影响外部平台（如加密钱包和Web3网站）的正确显示，并可能对代币价值产生不利影响。

Mutable Parameters

可变参数

ERC20 代币的 transfer() 函数通常包含开发者定义的参数，如交易税（taxFee）、流动性费（liquidityFee）等。这些参数通常在合约开发阶段由参与者共同决定。然而，若 Owner 拥有随意更改这些参数的权限，可能导致用户在不知情的情况下支付额外费用或遭受不公平交易。例如，PAWTH 是一个市值超过 100 万美元的 ERC20，合约 Owner 可以随意修改 taxFee、liquidityFee、burnFee 等参数。
